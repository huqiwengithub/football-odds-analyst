# 技能方法论诊断头脑风暴

## 定位

本技能**不是**赛后回测（那是 football-backtest-workflow 的职责）。本技能是**元诊断**：当一个回测报告暴露了亏损或异常后，系统性地检验投注技能方法论本身是否有关键盲区。

核心原则：
```
1. 所有发现 n=1 时 = 观察项，不是规则→必须跨赛事验证才能转化
2. 回测发现≠规则；验证三关: ≥2独立赛事 + 不降低已有准确率 + 不过度共线
3. 每次诊断必须同时列出支持和反对证据，禁止单项叙事
4. 根因绝不能停在单一维度——必须追问到多个子系统的交互
```

## 执行流程

### Phase 1: 诊断启动 → 暴露原始数据点 (Step 1)

输入：回测报告（HTML/JSON/文本）。读取后提取所有核心数字，不加任何解释。

**提取要求**:
- 方向准确率（投注场次/总场次）
- 实际 ROI 和绝对盈亏（¥）
- 核心亏损来源（具体哪一场 + 什么决策导致）
- 偏离信号触发情况（几场触发 / 真假阳性）
- 反事实最优方案（如果有）
- 其他异常（如 Kelly 归零、阈值排斥等）

**输出格式**: `N 个核心数据点`，纯数字描述，不附解读。每个数据点一句话。

### Phase 2: 多假设检验 (Steps 2-3)

#### Step 2: 每个数据点 ≥ 2 种解读

对 Phase 1 的每个数据点，至少生成两种完全不同的解读方向。格式：

```
数据点: [原文]
  解读 A: [一边的解读]
  解读 B: [另一边的解读，必须与 A 有明显分歧]
  (可选) 解读 C: [第三种不同视角]
```

**禁止**: 两个解读方向一致只是措辞不同的伪多元解读。

#### Step 3: 反向检验最舒服的结论

找出分析者直觉上最认同的那个结论（通常是"XX 没问题，只是 YY 需要微调"），然后**强制列出 ≥ 3 条反对它的证据**。

每条反对证据必须是具体的数据或逻辑推理，不能是"数据不够"这种空泛陈述。

### Phase 3: 三层归因 (Step 5)

对一个亏损/错误的归因不能停在表面描述。必须追问三层：

```
第 1 层: 直接原因 — "什么决策错了"
第 2 层: 系统原因 — "技能库的哪个模块/规则缺失导致了这个错误"
第 3 层: 结构性原因 — "这几个子系统是如何交互共谋产生这个错误的"
```

**关键**: 第 3 层必须以"跨子系统交互"为切入点。亏损通常不是单一模块的问题，而是阈值、仓位、信号、载体四者相互加强的结果。

### Phase 4: 统计自检 (Step 4)

对每个"发现"，逐项检查：

```
□ n >= 30? → 可做统计推断 / n < 30? → 仅观察项
□ 跨赛事验证三关:
   □ 是否 ≥ 2 个独立赛事验证过?
   □ 不会降低已有系统的准确率?
   □ 不与已有因子过度共线 (|r| < 0.6)?
□ 换分组方式（按日期/按赛事/按赔率区间），结论稳定吗?
□ 能用更简单的随机模型 (baseline 50%) 解释吗?
```

**铁律**: 不满足三关中任意一关的"发现"，只能标注为"观察项（待验证）"，不能写入技能规则。

### Phase 5: 生成开放问题 (Step 6)

所有诊断结论必须以开放性问题收尾。格式：

```
"我们发现的是 [陈述]，但还不清楚的是 [无法回答的问题]"
```

每个开放问题必须附带：
1. 验证路径（需要什么数据 + 用什么方法 + 预计耗时）
2. 最低验证门槛（需要多少样本才算有意义）
3. 可服用的现有数据（如果有）

### Phase 6: 输出诊断报告

整合 Phases 1-5 的输出，生成结构化诊断报告：

```
## 诊断摘要
[一段话概括核心发现]

## Phase 1: 原始数据点 (N 个)
## Phase 2: 竞争性解读矩阵
## Phase 3: 舒适结论的反向检验
## Phase 4: 三层归因图
## Phase 5: 统计自检清单
## Phase 6: 开放问题 + 验证路径
## 附录: 建议的技能更新点（按 P0/P1/P2 分级，标注验证状态）
```

## 附录 A: 技能盲区速查表

诊断时重点检查以下 4 个子系统是否各自或交互产生盲区：

| 子系统 | 文件 | 检查点 |
|:---|:---|:---|
| 赔率阈值 | betting-sop.md 1.5 | 低赔率场次是否被结构性排斥？ |
| 仓位/Kelly | betting-sop.md Kelly注码速查 | 低赔区 Kelly 是否误判 edge < 0？ |
| 信号独立性 | knowledge-base.md KB-20a | RQSPF/让球信号是否区分独立 vs 关联溢出？ |
| 投注载体 | betting-sop.md 第二步-A | SPF vs RQSPF 选择是否有系统规则？ |

## 附录 B: 跨赛事验证框架

当诊断产生假设后，按以下路径验证：

```
1. 服用现有数据集: WC2022 (64场) + EURO2024 (51场) + WC2018 (64场) = 179场
2. 分箱验证: 按赔率区间/让球深度/排名差等维度分箱
3. 跨赛事一致性: 同一发现在不同赛事中方向是否一致
4. 统计显著性: p < 0.05 + 置信区间不跨零
5. 边际贡献: 新因子在已有因子基础上的增量准确率
```

**验证通过条件**: 三个赛事中 ≥ 2 个方向一致 + p < 0.05 + 边际贡献 > 2pp。

## 边界

- 本技能只诊断方法论，不分析单场比赛——那是 football-backtest-workflow 的职责
- 本技能的输出是"假设 + 验证路径"，不是最终规则变更
- 任何未经跨赛事验证的假设都不能直接写入 knowledge-base.md / betting-sop.md
