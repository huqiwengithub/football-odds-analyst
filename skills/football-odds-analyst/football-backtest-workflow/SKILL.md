# 足彩技能回测工作流 v2.0.0

## 核心哲学

- **数据说话，不预设结论。** 报告每一层以数据为起点，不允许凭空提出概念框架。
- **头脑风暴法贯穿始终。** 对每个现象、每个数据点、每个假设，强迫自己生成 ≥2 种互相竞争的解读，保留解读分歧。
- **一次回测不产生规则。** 任何从单次回测中发现的"模式"都只能作为"观察项（待验证，n=1）"存在，不能升级为方法论规则。
- **结构固定。** 最终报告固定为 9 层，格式参考下面指定的 HTML 版本。

## 技能调用总览

```
Phase 1: 数据采集        → 调用 Skill → 500com-football-scraper
Phase 2: 赛前预测        → 调用 Skill → football-odds-analyst
Phase 3: 盈亏计算        → 调用 Skill → brainstorming（探索多种归因可能）
Phase 4: 多假设检验      → 使用 brainstorming 方法（已加载）
Phase 5: 信号拆解+决策审计 → 使用 brainstorming 方法（已加载）
Phase 6: 反事实推演      → 使用 brainstorming 方法（已加载）
Phase 7: 时间旅行验证    → 使用 brainstorming 方法（已加载）
Phase 8: 报告生成        → 使用 brainstorming 方法做最终检查
```

## 报告结构（固定 9 层）

最终输出的 HTML 报告必须严格遵循以下 9 层结构。这是经过验证的框架，不得删层、不得合并。

```
Layer 1:  原始数据（先看数字，不谈结论）
Layer 2:  多假设检验（不预设结论，逐个排除）
Layer 3:  关键信号拆解
Layer 4:  反事实推演（"如果当时…"的定量检验）
Layer 5:  决策节点审计（逐笔决策回溯）
Layer 6:  根因决策树
Layer 7:  时间旅行验证（知道错哪后，回到当时会怎么做？）
Layer 8:  基于数据的框架修改建议（非规则，为数据驱动的观察项）
Layer 9:  收敛结论
```

每一层的具体内容要求和执行方法见下。

## 7 阶段工作流

### Phase 1: 数据采集 — 先取比赛 ID，按 ID 查缓存，再抓取

> ⚠️ **遵循共享缓存协议**: 先取 shuju_id → 按 ID 查缓存 → 仅抓缺失场次。

具体操作：
```
Step 0: 调用 500com-football-scraper Phase 0 Step 0a → 获取 match_list（含 shuju_id）
  通过 Liansai API 或 trade.500.com 页面，一次轻量请求拿到所有比赛 ID

Step 1: 对 match_list 中每场的 shuju_id:
  检查 .cache/shared-football/parsed/{date}_{shuju_id}.json
  → 存在且未过期: 标记 CACHED，直接加载
  → 不存在或过期: 标记 NEED_FETCH，传入下一步

Step 2: 对 NEED_FETCH 的比赛:
  调用 500com-football-scraper Phase 0 Step 0c → 逐场逐页抓取
  → 每页先检查 raw/ 缓存（Page 1 不因 Page 2 缺失而重复抓取）
  → 解析后写入 .cache/shared-football/parsed/{date}_{shuju_id}.json

Step 3: 从 parsed/{date}_{shuju_id}.json 加载所有比赛数据
  确认 6 页数据完整（ouzhi/yazhi/rangqiu/daxiao/shuju/touzhu）
```

> 💡 **按 shuju_id 独立缓存的优势**：回测时可能只需要已完赛场次的数据，
> 不会因为某场未赛而影响其他场的缓存命中。之前抓过的比赛永久有效。```

### Phase 2: 赛前预测 — 调用 football-odds-analyst 执行 13 步流程

> ⚠️ 本技能不包含分析逻辑。调用 `Skill` 工具加载 `football-odds-analyst` 技能，严格按照其 SKILL.md 的 Step 0 → Step 12 执行 13 步流程。不得跳过任何步骤，不得用本 skill 的逻辑替代 analyst 的分析。

analyst 技能的 Step 0 会自动加载 scraper JSON 数据。执行完成后记录每场输出：
- 方向 + 把握度
- 仓位档位（🔥核心 / 🚫跳过 / 🔄冷门翻转）
- 偏离信号
- 串关组合 + 单关方案

---

### Phase 3: 盈亏计算

> 调用 `Skill` 工具加载 `brainstorming` 技能。在确定"最大亏损源"时，用 brainstorming 的方式探索多种可能性——先列出数据，再探索多种亏损归因，不急于下结论。

从 JSON 提取实际赛果，计算：

```
总投入:  ¥100
总回收:  ¥XX.XX
净利润:  ¥XX.XX
ROI:     X.X%

方向判断:   正确 X/总场次
核心方案:   中/未中
单关:       中/未中
最大亏损源:  [场次 + 原因]
```

---

### Phase 4: 多假设检验 → Layer 1 + Layer 2

> 🧠 **头脑风暴阶段**：brainstorming 技能已在上一步加载，此处深入应用。对每个数据点，按 brainstorming 方法的"探索多个角度、提出竞争性假设、不预设结论"原则执行。

#### Layer 1: 原始数据

呈现所有赔率数据、亚盘数据、基本面数据——不附解读、不附分析。

输出格式（3 张表）：
1. **赔率变动全景表**：场次/对阵/赛果/Pinnacle开盘→临场/变化量/亚盘初终/离散度/竞彩SPF
2. **亚盘详细数据表**（对核心失误场次）：公司/初盘/初盘水位/终盘/终盘水位/盘口变化
3. **原始特征提取**：列出值得注意的数据点，但不解释

#### Layer 2: 多假设检验

**执行方法（头脑风暴式）**：

对 Phase 3 确定的"最大亏损源"和"核心反直觉发现"，逐个生成假设。

**生成假设的规则**：
- 不得少于 5 个假设
- 每个假设必须基于具体数据点，不能凭空推断
- 对同一现象，探索完全相反方向的假设（例：既是"方向错了"也是"方向没错"）

**每个假设的格式**：
```
### H1: [假设名称] | [接受/拒绝/部分支持]
**陈述**：[假设内容]
**支持证据**：[数据点列表]
**反对证据**：[数据点列表]
**定量评估**：[具体数值计算]
**可信度**：[高/中/低]
```

**对每个假设的头脑风暴要求**：
- 写完"支持证据"后，必须专门去想"反对证据"——不能只找对自己有利的证据
- "定量评估"必须有具体数字，不能纯描述
- 可信度可以标注"n=1，统计上无效"

---

### Phase 5: 信号拆解 + 决策审计 → Layer 3 + Layer 5

> 🧠 继续应用 brainstorming 方法：对每一个信号，强迫探索 ≥2 种互相矛盾的解读。不允许只写一种解释。

#### Layer 3: 关键信号拆解

**执行方法**：
对每个被使用的信号（或应该使用但没使用的信号），做拆解表：

| 信号 | 来源 | 赛前可识别？ | 当时判断 | 事后验证 | 运用价值 |
|:---|:---|:---:|:---|:---:|:---:|

对其中 1-2 个核心信号，额外做**深入分析**：
- 列出该信号的所有可能解释（头脑风暴：≥2 种）
- 如果可能存在"共识同步"现象（≥80%公司同步调整），列出市场资金驱动 vs 信息驱动两种可能性
- **不要选一种解释作为正确答案，保留多种可能性**

#### Layer 5: 决策节点审计

对每个关键决策节点（入核心/跳过/单关），做回溯表：

```
决策: [入/跳过 某场]
依据: [当时依据的数据]
替代选项: [其他可能决策]
审计: [好决策/有盲区/错误]
```

---

### Phase 6: 反事实推演 + 根因决策树 → Layer 4 + Layer 6

> 🧠 反事实推演本身就是一种 brainstorming 方法——"如果当时选了另一条路会怎样"。用 brainstorming 的"探索替代方案"原则，生成 ≥4 种不同风险偏好的反事实方案。

#### Layer 4: 反事实推演

**执行方法**：
保持方向判断不变，改变仓位结构和串关方式，模拟 ≥4 种替代方案。

每种方案必须有：
```
方案名称: [描述]
具体仓位: [各注金额+赔率+组合]
总回收: ¥XX
净利润: ¥XX
ROI: X%
vs 实际方案: ±¥XX
```

反事实方案不能只写"理想的"，要覆盖不同风险偏好的方案（保守型、稳健型、激进型各至少 1 个）。

**汇总表**：对比所有方案的实际结果。

#### Layer 6: 根因决策树

从结果出发，逆向追踪每条因果链，标注各节点贡献的亏损量。

格式：
```
净利润 -¥71.80
  ├─ 核心2串1（¥80）全赔
  │   ├─ 串A ¥40: 德国✅ × 厄瓜❌ → ¥0
  │   └─ 串B ¥40: 德国✅ × 厄瓜❌ → ¥0
  └─ 辅助单关 ¥20: 荷兰✅ + 日本✅ → ¥28.2
```

底层归因分解（加星号标注"n=1 非统计有效"）：
```
仓位集中: XX%  *
共识盲区: XX%  *
基本面遗漏: XX%  *
随机性: XX%    *
```

### Phase 7: 时间旅行验证 → Layer 7

> 🧠 时间旅行模拟是 brainstorming 的"假设思维"应用——在思维中构建一个平行世界，探索不同的决策路径对应的结果。

**核心问题**：站在赛前，只知道改进后的思维方法（不知道赛果），重新做决策，结果会更好吗？

#### Step 1 — 逐场重新评估
对每场比赛，用原始数据 + **从本次回测中学到的多种可能解读** 重新判断。

不是"用更好的框架"，而是"这次我知道了同一个数据可以有多种读法，在这个前提下重做"。

| 场次 | 原始决策 | 考虑多解读后的新判断 | 变/不变 |
|:---|:---:|:---:|:---:|

#### Step 2 — 重建仓位方案
按新的多解读决策，重建仓位结构。

#### Step 3 — 计算新方案损益
| 指标 | 原方案 | 新方案 | 差异 |
|:---|:---:|:---:|:---:|
| ROI | X% | X% | ±Xpp |

#### Step 4 — 诚实警示
```
时间旅行模拟存在回溯偏差——在"知道哪些解读可能有价值"的前提下做决策。
真正的验证需要≥5次前瞻性应用。在此之前结论不具统计意义。
```

### Phase 8: 报告生成 → Layer 8 + Layer 9

> 🧠 最后检查：报告中的"观察项"和"收敛结论"是否经过了 brainstorming 验证？ Layer 9 以开放性问题结尾——这也是 brainstorming "不预设结论"原则的体现。

#### Layer 8: 基于数据的框架修改建议（观察项，非规则）

对本次回测中发现的"可能值得注意"的数据模式，以观察项形式列出。

每个观察项的格式：
```
### 观察项 O1：[名称]
**数据来源**：[具体数据点]
**草案**：[可能的含义]
**样本量**：1个赛事，n=1。需验证。
```

关键约束：
- 每个观察项必须对应具体数据点，不能来自直觉
- 每个必须标注"n=1，需验证"
- 不能写成"规则"——用"可能/值得关注/有待验证"等语言
- 数量控制在 4 个以内

#### Layer 9: 收敛结论

收敛结论不是"拍总结"，而是从数据中提取的 3-5 个具体陈述。

每个结论必须：
1. 引用具体数据支撑
2. 标注统计局限（如果适用）
3. 不做超出数据范围的推断

格式示例：
```
1. 仓位集中是本次亏损的直接原因（反事实推演：4种替代方案全部盈利）
2. 方向判断准确率75%，在n=4样本下统计上等同于随机（不能说明任何问题）
3. 深盘场（赔率<1.25）+离散度<5的"极端共识"场景，在本次回测中出现了与共识相反的赛果
   → 开放性问题：这只是一个巧合，还是值得进一步观察的模式？
```

**Layer 9 必须以开放性问题结尾**。不能做出"我们已经找到了答案"的姿态。

### Output Language — 报告必须小白能看懂

> ⚠️ **9 层报告的所有文字必须用小白都能看懂的语言**。
> 庄家名字（Pinnacle、bet365等）保留原名，其余全换白话。

#### 核心原则

```
1. 所有英文缩写（MBI、OCI、SPF、AH、OU、EV、ROI等）→ 按下方翻译表写成大白话
2. 所有内部术语（离散度、水位、初盘、抽水等）→ 按下方翻译表写成大白话
3. 庄家公司名字保留原名（Pinnacle、bet365、威廉希尔等）
4. 数值用大白话标注含义（eg. "1.92倍回报率"而非"赔率1.92"）
5. 金额用 ¥ 展示，不出现 $ 或 decimal only
6. 每层报告的标题和结论段落必须能用一句话说完
```

#### 术语翻译对照（必须遵守）

| 内部术语 | 回测报告中必须写成 |
|:---|:---|
| Pinnacle | Pinnacle（保留，可加注"最精明的博彩公司"） |
| bet365 / 威廉希尔 | 保留原名 |
| ROI | 收益率 |
| EV (期望值) | 预期回报 |
| 赔率 | 回报倍率 |
| 水位 | 回报率含的水分 |
| 初盘 | 开盘时的赔率 |
| 临场/终盘 | 开赛前的最终赔率 |
| 升水 | Pinnacle调高了某方的回报率（不太看好） |
| 降水 | Pinnacle调低了回报率（看好） |
| 离散度/离散收敛 | 30家博彩公司的意见统一了 |
| 离散发散 | 30家公司吵起来了、分歧加大 |
| 抽水 | 博彩公司从中抽走的利润 |
| 去抽水 | 扣除博彩公司利润后的真实概率 |
| 核心仓位 | 最有把握的投注 |
| 跳过/不投 | 没把握，先不买 |
| 冷门翻转 | 系统提示可能爆冷，建议反着看 |
| 量价背离 | 赔率和下注资金走向不一致 |
| 假设检验 | 逐一验证各种可能性 |
| 反事实推演 | "如果当时选了另一条路会怎样"的模拟 |
| 根因决策树 | 从亏损结果反向追溯各环节的影响 |
| 时间旅行验证 | 知道结果后，重新回到赛前做决策 |
| 仓位集中 | 把钱太集中地压在少数比赛上 |
| 共识盲区 | 大家意见太一致反而忽略了风险 |
| 随机性 | 纯粹运气因素，无法预测 |

#### 常用缩写翻译速查

报告中遇到这些缩写时，按以下翻译写成大白话（不用禁止，但要翻译）：

```
MBI → 多家博彩公司综合信号
OCI → Pinnacle 的赔率变动方向
SPF → 胜平负
AH  → 让球盘（强队让弱队）
OU  → 大小球 / 总进球数
EV  → 预期回报
ROI → 收益率
Kelly → 仓位比例公式
Shin → 去抽水算法
```

> 💡 **原则**：缩写可以出现在报告中，但必须是翻译后的大白话形式。
> 庄家名字（Pinnacle、bet365等）保留原名不翻译。

#### 每层语言检查

```
Layer 1 (原始数据):  表格标题用中文，"Home"→"主队"、"Away"→"客队"
Layer 2 (多假设检验): 假设陈述用"如果…那么…"句式，不要用书面语
Layer 3 (信号拆解):  每个信号用一句话解释"这个信号当时告诉我们什么"
Layer 4 (反事实推演): 方案名称用"保守方案/中性方案/激进方案"而非"A方案/B方案"
Layer 5 (决策审计):   决策评价用"好决策/还行/有盲区/判断错误"
Layer 6 (根因决策树): 每层归因用中文数字标注，"XX%来自XX原因"
Layer 7 (时间旅行):   用"如果当时知道这些，我们可能会…"的假设语气
Layer 8 (观察项):     每条观察项先写一句话结论，再跟数据说明
Layer 9 (收敛结论):   结论用"我们发现…"句式，开放性问题用"但还不清楚…"
```
